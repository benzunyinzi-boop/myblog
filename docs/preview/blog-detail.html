用 errgroup 优雅地编排并发任务

● 2026.04.28 ◷ 阅读 8 分钟 ≡ 2,431 字 #go #concurrency #errgroup

在 Go 里写并发,goroutine + channel 是基本功;但一旦任务数变多、彼此又有依赖,代码很容易就堆成一锅 sync.WaitGroup 加一堆 error channel 的大杂烩。 今天聊聊我日常最常用的一个工具:golang.org/x/sync/errgroup。

errgroup 不是魔法,它只是把 WaitGroup、error 汇聚、context 取消这三件事封在了一起。 理解它,也就理解了并发编排中最常见的三种诉求。

WaitGroup 的痛点

最早我们都是这样写的:起几个 goroutine,用 sync.WaitGroup 等它们跑完, 再用一个 error channel 把错误收集起来。问题是,一旦其中一个失败了, 我们往往希望其他任务也立刻停下来,但 WaitGroup 本身不提供取消能力。

典型的"老代码"

func FetchAll(ctx context.Context) error {
    var wg sync.WaitGroup
    errCh := make(chan error, 3)

    for _, id := range ids {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            if err := doFetch(ctx, id); err != nil {
                errCh <- err
            }
        }(id)
    }
    wg.Wait()
    close(errCh)

    // 第一个错误抛出去,其余吞掉
    for err := range errCh {
        if err != nil { return err }
    }
    return nil
}

看起来没什么大问题,直到线上某一天 doFetch 某个请求挂在远程不返回, 其他 2 个子任务早就跑完了,但 wg.Wait() 仍在苦苦等待。 这是典型的"缺少取消联动"。

errgroup 的做法

errgroup.WithContext 返回一个 group 和一个派生的 ctx。 只要有任何一个 Go() 返回了非 nil 错误,派生 ctx 就会被取消, 其他 goroutine 只要正确尊重 ctx,就能被快速打断。 

import "golang.org/x/sync/errgroup"

func FetchAll(ctx context.Context) error {
    g, gctx := errgroup.WithContext(ctx)
    for _, id := range ids {
        id := id // 捕获变量
        g.Go(func() error {
            return doFetch(gctx, id)
        })
    }
    return g.Wait()
}

代码从 20 行瘦到 8 行,并且自动获得了:

  • 任意失败即取消的联动;
  • g.Wait() 返回第一个非 nil 错误;
  • 对 doFetch 没有侵入 —— 它只要是一个"返回 error 的函数"即可。

生产事故回溯

去年有一次线上 p99 延迟突然飙高,排查发现是一个"聚合接口"里, 三路并发调用其中一路超时了,但 WaitGroup 没取消另两路, 另两路在 5s 后才各自超时返回 —— 一次请求因此多消耗了 4s 时间。 切换到 errgroup 后,第一个超时后其余 goroutine 跟着 ctx 取消, p99 从 6.2s 降到 1.1s。

这件事的教训是:不要让"并发"变成"并发等待"。 任何长生命周期的 goroutine 都必须有一条明确的退出通路。

进阶:限制并发数

errgroup 1.5+ 支持 SetLimit,可以在不引入 semaphore 的前提下限制并发: 

g, gctx := errgroup.WithContext(ctx)
g.SetLimit(16) // 最多 16 个同时在跑

for _, url := range urls {
    url := url
    g.Go(func() error {
        return crawl(gctx, url)
    })
}
return g.Wait()

小结

errgroup 并不能解决所有并发问题,但它把 90% 的"并发 + 错误 + 取消"场景封装得足够好。 对于更复杂的场景,比如"部分任务失败可以容忍"、"需要分组重试", 你依然需要自己动手造轮子 —— 但先把 errgroup 用熟,再决定什么时候该跳出来。
